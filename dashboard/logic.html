MOMENTUM BOT LOGIC

추천 로직 설명

후보군 생성부터 검증/보정/모델 확장까지 전체 흐름을 정리했습니다.

대시보드 추천 로직 운영 상태

1) 후보군 생성

빗썸 현물 + 비트겟 USDT-M 선물의 공통 심볼만 사용합니다.

롱: 빗썸/비트겟 24h 상승률 기준 숏: 비트겟 24h 하락률 + 펀딩 조건 유동성 바닥: 빗썸 거래대금 + 비트겟 거래량 출력 수량: top=3, SHORT 최소 1개(후보 존재 시)

2) 리스크 필터

과열/극단 펀딩/주문불가 종목을 제거해 변동성 리스크를 줄입니다.

과열 상한 보수 상한 절대 펀딩 상한 빗썸 주문 가능 여부 손실 경고 종목 쿨다운 차단

3) 모델 구조

  • v1: 기본 모멘텀/유동성/펀딩 점수
  • v2: 시장 추세(24h~1m), 펀딩 crowding, OI, 쏠림(regime) 보강
  • 활성 모델은 model_registry로 제어

4) 검증과 승률 집계

  • 추천 시점의 진입가를 저장
  • horizon 경과 후 blended 수익률 평가
  • SHORT는 비트겟 수익률을 우선 평가(없을 때만 빗썸 대체)
  • 양 거래소 데이터가 모두 비면 horizon 1회 추가 유예 후 확정
  • 모델별 count/win_rate/avg/median을 별도 집계

5) 자동 보정

  • 저성과 시 보수적으로 필터 강화
  • 후보 고갈 시 완만하게 필터 완화
  • 보정 악화 시 롤백 + 6시간 쿨다운
  • 롤백 조건: 최근 보정 이후 승률 델타가 -5.0%p 이하

6) 자동 모델 확장

  • 조건: 모델 평가수 24 이상 + 승률 45% 미만
  • LONG: momentum_long_v1 → momentum_long_v2
  • SHORT: momentum_short_v1 → momentum_short_v2
  • 확장 이벤트는 run_history/model_governance_events에 기록

7) 시장 상관도 해석 기준

  • 시장 지표는 24h/12h/6h/1h/15m/5m/1m 다중 타임프레임 변화율을 사용합니다.
  • 시장 쏠림(regime)은 btc/eth/single-alt/alt-broad/balanced로 분류합니다.
  • 시장/BTC/ETH 추천상관은 추천방향(LONG=+1, SHORT=-1)과 시장 변화(1h 우선, 없으면 24h)의 상관계수입니다.
  • 따라서 ETH 추천상관은 ETH vs ETH 자기상관이 아니라, 추천 방향과 ETH 장세의 정합성을 뜻합니다.
  • 예: ETH 추천상관이 -0.17이면 최근 추천 방향이 ETH 장세와 약하게 반대로 움직였다는 의미입니다.
  • 표본(n)은 신뢰도 판단을 위해 유지합니다. n이 작으면 상관값 변동성이 커집니다.

8) 누적 저성과 시 모델 추천

  • 트리거: 최근 검증 표본 24건 이상 + 승률 48% 또는 평균수익률 -0.10%.
  • 시장 적합도: 현재 후보군에 대해 모델별 점수(상위 평균)를 계산해 시장 적합도를 산출합니다.
  • 과거 성과: 모델별 win_rate/avg_return/count를 신뢰도 가중으로 반영합니다.
  • 출력: LONG/SHORT별 유력 모델과 근거(fit edge, 과거 승률, 표본)를 ModelHint로 텔레그램/대시보드에 표시합니다.

9) 손익비(R:R) 정의

  • 방향 정의: LONG은 +1, SHORT는 -1 방향 계수를 사용합니다.
  • 변동성 입력: vol24 = clamp((|b_rate24h| + |g_rate24h|)/2, 0.5, 25.0)
  • 손절폭(%): stop_pct = clamp(0.45 + vol24*0.08, 0.45, 2.80)
  • 기본 손익비 배수: rr_base = clamp(1.10 + score*1.50, 1.10, 2.60)
  • 목표폭(%): target_pct = stop_pct * rr_base
  • 진입 오프셋(%): entry_offset_pct = clamp(0.10 + vol24*0.02, 0.10, 0.80)
  • 진입 추천가: entry_reco = now_price * (1 - direction*entry_offset_pct/100)
  • 시점 손익비(rr_now): 현재가 기준 |target-now| / |now-stop|
  • 진입 손익비(rr_entry): 추천 진입가 기준 |target-entry| / |entry-stop|
  • 표시 기준: 대시보드에는 시점가(KRW/USDT), 시점 손익비, 진입 추천가(KRW/USDT), 진입 손익비를 함께 표기합니다.

10) 손실 경고와 재추천 차단

  • 보유 중인 pending 추천을 먼저 실시간 재평가하고 손실 경고를 탐지합니다.
  • 경고 기준: live_return_blended < loss_alert_threshold (기본 0.0, 손실 전환 시).
  • 경고가 난 심볼은 loss_cooldowns에 등록되어 기본 60분 동안 후보군에서 제외됩니다.
  • 손실 탐지는 추천 전/후 2회 수행되어, 경고 직후 같은 심볼 재추천을 최대한 방지합니다.

11) 실행 주기와 즉시성

  • 정기 추천 스캔: GitHub Actions cron 기준 5분마다 1회 실행.
  • 손실 워치독: 각 스캔 직후 --alerts-only --watch --interval-sec 15 --cycles 14로 약 3.5분 감시.
  • 워치독 모드에서는 신규 추천 없이 손실 경고/검증만 수행해 알림 지연을 줄입니다.

12) 텔레그램 출력 규칙(컴팩트)

  • 추천 라인: 심볼 | LONG/SHORT | score | funding | OI 형식으로 전송합니다.
  • 손실 발생 시 별도 경고 메시지(종목/방향/실시간 손익률)를 추가 전송합니다.
  • 쿨다운으로 제외된 종목 수는 Risk block 라인으로 함께 안내합니다.

13) 문서 동기화 원칙

  • 추천/검증/보정/알림 로직 변경 시 본 페이지(docs/logic.html)를 같은 커밋에서 동시 업데이트합니다.
  • 대시보드 노출용 문서(dashboard/logic.html)도 동일 내용으로 동기화합니다.
